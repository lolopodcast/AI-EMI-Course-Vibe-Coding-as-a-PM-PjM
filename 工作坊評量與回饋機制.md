# 工作坊評量與回饋機制

> 對應：[工作坊設計文件](工作坊設計_AI-native產品開發工作坊.md)（第 2 節學習目標 LO1-LO9）
> 設計原則：這是一場**技能導向**的工作坊，評量重點應該放在「有沒有做出東西、做出來的東西夠不夠具體」，而不是紙筆測驗。全程用「作品／產出物」作為評量證據，搭配結構化的課後回饋問卷。

---

## 1. 學習目標對照評量表

| 編號 | 學習目標 | 評量方式 | 證據（產出物） | 檢核時機 |
|---|---|---|---|---|
| LO1 | 說出六層架構，解釋順序為什麼重要 | 口頭提問／小組互評 | 一句話專案宣言 | Session A1 尾聲 |
| LO2 | 寫出 Needs → Goals → Open Spec | 講師巡場檢查＋同儕互評 | Needs/Goals/Spec 文件 | Session A2-A3 |
| LO3 | 畫出 Graphiti（純文字關聯樹） | 講師巡場檢查 | 關聯樹 | Session A4 |
| LO4 | 設計 Schema | 講師巡場檢查 | 資料規格表 | Session A4 |
| LO5 | 說出 Agile 概念並對應到六層架構 | 小組口頭問答＋看板產出 | 初版看板 | Session A5 |
| LO6 | 用 Claude Code 做出可運作原型 | 產出物檢視 | 原型＋Vibe Coding Log | Session B2 |
| LO7 | 判斷任務平行/相依，會用多重代理人 | 講師巡場檢查 | 更新後的 Vibe Coding Log | Session B3 |
| LO8 | 寫出可獨立使用的交接文件 | **同儕實測**（見第 3 節） | 交接文件 | Session B4-B5 |
| LO9 | 主持一場檢討流程而非檢討人的回顧會 | 觀察＋自評 | Keep-Problem-Try 紀錄 | Session B6 |

👉 核心原則：**LO8 是全場最關鍵的檢核點**——用「另一組完全沒參與過程的人，能不能只靠文件接手」直接驗證，比任何問卷都準。

---

## 2. 各 Session 檢核點（給講師巡場用的簡易檢查表）

| Session | 檢核問題（講師巡場時自問） |
|---|---|
| A2 | 這組的 Goals 是「結果指標」還是「功能清單」？ |
| A3 | 這組的使用者故事，讀起來 AI 會不會需要再追問？每條都有 DoD 嗎？ |
| A4 | 關聯樹裡的每一條線，Schema 都有對應欄位嗎？ |
| A5 | 看板上的卡片，是不是都能對到 Open Spec 裡的某一條？ |
| B2 | 原型跑出來後，S1-S5（或該組自己的故事）逐條核對 DoD，有沒有落差？ |
| B3 | 任務排序有沒有考慮到相依關係，還是照便利貼順序硬做？ |
| B5 | 交接文件被拿走後，對方卡住的地方，是不是文件裡真的沒寫清楚？ |

## 3. 結業標準（Definition of Done，給整場工作坊用）

完成以下三項，即符合結業標準，可核發**工作坊結業證明／研習時數證明**：

1. ✅ 一份完整的**基礎規格包**（Needs／Goals／Open Spec／Graphiti／Schema／看板）
2. ✅ 一個**可運作的原型**＋對應的 **Vibe Coding Log**
3. ✅ 一份**交接文件**，並且**通過同儕實測**——另一組只憑這份文件（不能問本人），實際延伸出至少一個新功能

👉 第 3 項是刻意設計的「硬門檻」：因為這整套工作坊的核心主張就是「規格與文件必須自足」，如果連隔壁組都看不懂，代表這位學員還沒有真正掌握 LO8。

## 4. 課後回饋問卷設計

建議工作坊結束當天，用 Google 表單／紙本皆可，控制在 5 分鐘內填完。

### Part A｜學習成效自評（前測／後測對照，1-5 分量表）
> 請針對每一項，回想「工作坊開始前」與「現在」，各給自己打一個分數（1 = 完全沒信心，5 = 非常有信心）

| 項目 | 開始前 | 現在 |
|---|---|---|
| 我能說出 AI-native 六層架構，並解釋為什麼順序重要 | ___ | ___ |
| 我能自己寫出一份清楚的專案規格（Needs/Goals/Spec） |___ | ___ |
| 我能用 Claude Code 把規格變成一個可運作的原型 | ___ | ___ |
| 我能寫出一份讓別人不用問我就能接手的交接文件 | ___ | ___ |
| 我知道怎麼帶一個小型團隊做 AI 專案（含 Agile 節奏） | ___ | ___ |

### Part B｜內容與節奏（1-5 分量表 + 開放題）
1. 基礎模組（Needs/Goals/Spec/Graphiti/Schema/Agile）的份量與節奏，會不會太快／太慢？（1 太慢 – 5 太快，3 為剛好）
2. 實施模組（Vibe Coding/多重代理人/交接演練）的份量與節奏，會不會太快／太慢？
3. 哪一個 Session 對你最有幫助？為什麼？（開放題）
4. 哪一個 Session 最需要改進？為什麼？（開放題）

### Part C｜講師與教材
1. 講師講解是否清楚易懂（1-5）
2. 案例（駱教授 EMI 課程）是否有助於理解（1-5）
3. 交接文件範本／示範腳本等教材，是否會想帶回去實際使用（是／否／部分會）

### Part D｜後續應用意願
1. 你是否已經有具體想套用這套流程的專案？（是／還在想／沒有）
2. 如果是，你預計會先自己做，還是會找 TA／同事一起組隊？
3. 是否願意在 30 天後，接受一次簡短的追蹤訪談／填一份追蹤表單？（是／否）

---

## 5. 短期追蹤機制（選用，建議正式開課後採用）

給結業後 30 天的簡短追蹤（一封 email + 3 題表單即可）：

1. 你有沒有實際開始套用這套流程在自己的專案上？（有／還沒開始／已經放棄）
2. 如果有，你走到六層架構的哪一層了？
3. 過程中卡住的地方是什麼？（開放題，可作為下一梯次課程修正的依據）

👉 這份追蹤不只是關心學員,也是最直接的「這套工作坊教材有沒有真的教會人」的驗證——比課堂當下的滿意度分數更有參考價值。

---

## 6. 給講師的回饋整理建議

- **每梯次結束後**，把 Part A 的前測／後測分數整理成一張長條圖，五個項目分別看「進步幅度」，進步幅度最小的那一項，就是下次教材要優先調整的地方
- **開放題（Part B 第 3、4 題）**建議直接摘錄關鍵字，累積 2-3 梯次後找重複出現的詞——這通常比單一梯次的個別意見更值得採納
- **結業標準第 3 項（同儕實測）失敗的案例**，務必記錄下來當作下一梯次 Session B4/B5 的反例教材，比講師自己編的反例更有說服力
