## 1. 案例基本設定（開場秀出，5 分鐘）

**人物**：駱世民教授（Prof. Shihmin Lo），教授一門 EMI（全英語授課）課程「國際策略管理（國策管）／ Strategic Management of Multinational Corporations（SMMC）」，修課學生約 60 人，大三、大四為主。

**背景痛點**：
- 學生反映全英語授課，課後常常看不懂教材內容，也不太敢舉手發問
- 駱教授沒有足夠時間逐一回覆學生課後的重複性問題
- 系所希望能追蹤學生在哪些單元卡關，作為未來課程調整的依據

**想做的東西**：一個課程網站，附帶一個雙語（中／英）AI 助教，能回答學生關於課程內容的問題，並記錄下每次問答，讓老師事後能檢視學生卡在哪裡。

## 2. 本案例的完整基礎規格包（可直接複製使用）

### Customer Needs
- 學生反映 EMI 全英語授課，課後常常看不懂教材、不敢發問
- 老師沒有足夠時間逐一解答學生的重複性問題
- 系所希望追蹤學生在哪些單元卡關，作為課程調整依據

### Product Goals
- Goal 1：建立課程入口網站——80% 學生能在開學兩週內找到教材與大綱
- Goal 2：提供雙語 AI 助教問答功能——每週至少 50% 修課學生使用一次
- Goal 3：產出「單元卡關」報表——老師每週能看到最多學生提問的前三個主題

### Open Spec（使用者故事 + 完成的定義）
| 編號 | 使用者故事 | 完成的定義（Definition of Done） |
|---|---|---|
| S1 | 身為學生，我想要在課程網站上看到每週的教材與大綱，以便提前預習 | 首頁列出 16 週課程大綱，每週有對應教材下載連結 |
| S2 | 身為學生，我想要用中文或英文問 AI 助教關於課程內容的問題，以便下課後也能得到解答 | 聊天視窗可輸入問題，AI 回覆需引用該週教材內容，中英文皆可提問並以提問語言回覆 |
| S3 | 身為老師，我想要每次學生與 AI 助教的對話都被紀錄，以便之後檢視 | 每筆對話含時間戳記、學生代號（匿名）、問題、AI 回覆，可匯出成表格 |
| S4 | 身為老師，我想要看到每週最常被問的三個主題，以便調整下週上課重點 | 報表頁面依提問次數排序，列出前三名主題與代表性問句 |
| S5 | 身為學生，我想要 AI 助教在不確定時明確告知，而不是隨意作答，以便我知道該去問老師 | 當問題超出教材範圍或信心不足時，AI 必須明確回覆「這個問題教材沒有涵蓋，建議詢問老師」，不得自行編造答案 |

### Graphiti（物件關聯，縮排文字樹）
```
Course（國際策略管理／SMMC）
 ├── Module（每週單元，共 16 週）
 │     └── Material（教材檔案／連結）
 ├── Student
 │     └── Conversation（與 AI Coach 的對話紀錄）
 ├── AI Coach
 │     └── Conversation
 └── WeeklyReport（依 Conversation 統計而成）
```

### Schema（資料規格表）
**Course**：id、title、semester、description
**Module**：id、course_id、week_number、topic、material_url
**Student**：id、alias（匿名代號）、email
**Conversation**：id、student_id、module_id（可為空）、question、answer、language、confidence_flag、timestamp
**WeeklyReport**：week_number、top_topics（清單）、question_count

## 3. 逐分鐘示範流程（90 分鐘）

### 段落 1（0–10 分鐘）暖身：秀出完整規格包
**講師說**：
> 「這是駱教授在基礎模組已經寫好的規格包。我們接下來要做的事，就是把這整份文件直接交給 Claude Code，看它能不能做出一個可以用的原型。」

投影第 2 節的完整內容，快速導覽五層各自對應到哪一段，提醒學員：這五層是他們昨天／上午已經親手寫過的東西，今天只是換一個工具去執行它。

---

### 段落 2（10–20 分鐘）組合第一次的完整指令

講師現場貼上以下指令給 Claude Code（可直接複製使用）：

```
我要開發一個 EMI 課程網站，附帶雙語 AI 助教功能。以下是完整規格，請你先讀過一遍，
用三到五句話跟我確認你的理解是否正確，再開始實作一個可以在瀏覽器打開的原型
（先用單一 HTML 檔案即可，不需要串接真實資料庫或真實 AI API，重點是先看得到畫面與互動邏輯）。

【Customer Needs】
（貼上第 2 節 Needs）

【Product Goals】
（貼上第 2 節 Goals）

【Open Spec（使用者故事 + 完成的定義）】
（貼上第 2 節 Open Spec 表格）

【Graphiti（物件關聯）】
（貼上第 2 節關聯樹）

【Schema】
（貼上第 2 節資料規格表）
```

**講師強調的重點**：先請 Claude Code「用自己的話確認理解」再動手，這等同於團隊裡的需求覆核——先抓出誤解，再動工，會比做完才發現方向錯了省時很多。

---

### 段落 3（20–40 分鐘）觀察第一輪回應與產出

**講師說明 Claude Code 通常會做的事**（依現場實際情況調整）：
1. 先用幾句話摘要它讀到的需求
2. 可能會提出 1–2 個釐清問題，例如：「教材要不要先假設是純文字內容？」「AI 助教的回覆內容，這一版是要真的接 AI，還是先用模擬回覆示範互動邏輯？」
3. 開始建立檔案，說明它打算怎麼組織這個原型
4. 產出一個可以打開預覽的原型（例如一個 HTML 頁面，含 16 週大綱列表、雙語問答聊天視窗、老師端簡易報表頁籤）

**講師示範如何回答釐清問題**（示範「規格不夠具體時當場怎麼補」）：
> 「教材先假設是文字內容沒關係，重點是問答邏輯要能示範出來；AI 助教這一版先用模擬回覆就好，不用真的接 API，但請保留『信心不足時要明確告知』這條邏輯，這是 S5 的重點。」

### 段落 4（40–55 分鐘）對照組演示：故意用「模糊規格」重來一次

**講師說**：
> 「現在我們故意示範一次，如果規格寫得很模糊，Claude Code 會做出什麼樣的結果。」

改用一個刻意模糊的指令：

```
幫我做一個課程網站，有 AI 可以回答學生問題。
```

觀察並記錄這一版通常會缺漏什麼，例如：
- 沒有雙語（因為沒說要中英文）
- 沒有對話紀錄（因為沒說要記錄）
- 沒有「信心不足要明確告知」的邏輯（因為沒說不能亂回答）
- 沒有每週報表（因為沒說要追蹤卡關主題）

**帶討論的問題**：
> 「這些缺漏，是不是剛好都對應到完整規格包裡『有明確寫出來』的那幾條？」

👉 這是整堂課最關鍵的體驗時刻：讓學員親眼看到「規格清楚 vs 模糊」直接決定了 Claude Code 產出的品質，而不是聽講師用嘴巴講。

---

### 段落 5（55–85 分鐘）各組操作：換成自己的規格包

各組把自己在基礎模組寫好的規格包，依段落 2 的指令格式貼給 Claude Code。

**巡場提醒重點**：
- 讓 Claude Code 先複述理解，再動手
- 每完成一個使用者故事，就對照它的完成的定義打勾
- 遇到落差時，先回頭修規格再重新下指令，而不是不斷重複同一句話
- 開始填寫交接文件範本第 7 節「Vibe Coding Log」

---

### 段落 6（85–90 分鐘）收尾

- 每組用一句話分享：「規格哪一條寫得最清楚？」「哪一條 Claude Code 理解錯了？」
- 講師預告 Session B3（多重代理人）：
  > 「如果這個專案要同時做『課程頁面』『AI 助教問答』『每週報表』三件事，下一節課我們會示範怎麼把這三件事拆給不同的子代理人同時處理。」

## 4. 常見狀況與講師應對（FAQ）

| 狀況 | 建議應對 |
|---|---|
| Claude Code 產出跟預期落差很大 | 回頭檢查是哪一層規格不夠具體（通常是 Open Spec 或 Schema），不要直接重下同樣的指令 |
| 學員想要「一次就做到完美」 | 提醒最小可行產品（MVP）精神——先求有能動的原型，細節之後再補 |
| 學員問「這樣做出來的東西可以直接上線用嗎」 | 澄清：工作坊產出是「原型」，正式上線前還需要工程審查、資安檢查、真實資料串接等，不在本工作坊範圍內 |
| 學員擔心自己的規格包不夠好 | 提醒：規格包本來就會隨著看到 Claude Code 的實際產出而修正，這正是 Graphiti↔Schema 疊代精神的延伸——規格不必一次到位 |

## 5. 本案例最終可能的原型形式（供講師參考）

一個單一 HTML 檔案，包含：
- 課程 16 週大綱列表區塊，每週附教材連結
- 雙語（中／英）問答聊天視窗（先用模擬回覆示範互動邏輯，不強制真的串接 AI API）
- 老師端簡易報表頁籤，顯示模擬的「本週最常被問三個主題」

👉 這與互動式課程教材網頁的產出型態一致，代表工作坊產出的原型，後續也可以真的發展成正式的課程教材網頁，而不只是課堂練習用的範例。
