# Session B5 自學指南：一人自學版——交接文件測試

> 對應：[工作坊設計文件](工作坊設計_AI-native產品開發工作坊.md) Session B5（原設計 90 分鐘，現場版本需要另一組人交換文件；本自學版改用「全新 Claude Code 對話」取代真人，抓 60-70 分鐘）
> 前置作業：完成 [SessionB4自學指南](SessionB4自學指南_交接文件撰寫.md)，手上有一份填好的交接文件
> 本文件風格：直接對你（自學者）說話——這是整套自學教材裡，唯一需要「換一種方法」而不是單純照抄現場版本的一節

---

## 學習目標

完成本節後，你應該能夠：
- 客觀驗證自己的交接文件是否真的「自足」，不是自己覺得清楚就好
- 用一個全新、沒有任何記憶的 Claude Code 對話，模擬「完全沒參與過程的人」
- 從測試結果，精準定位文件裡哪一節寫得不夠具體

---

## 0. 為什麼自學版要換方法

現場工作坊的 Session B5，是讓兩組學員互換交接文件，只憑文件接手對方的專案。**自學情境下沒有第二組人**，但這剛好給了一個更客觀的替代方案：

👉 **一個全新開啟、沒有讀過你任何對話紀錄的 Claude Code，就是最誠實的「完全沒參與過程的人」**——它不會因為「大概猜得到你的意思」而放你一馬，也不會像真人一樣礙於同學情面不好意思說「這裡看不懂」。它只會照文件字面上寫的去做，寫得不夠清楚，它就會做錯或一直追問。

---

## 1. 準備一個「乾淨」的測試環境（10 分鐘）

**重點：這個新對話絕對不能看得到你原本的專案脈絡**，否則測試沒有意義。

1. 在電腦上另外新增一個空白資料夾，例如 `交接測試`（跟你原本 Session B1-B4 用的資料夾完全分開）
2. 打開終端機，切換到這個新資料夾
3. 啟動一個全新的 `claude` 對話（如果你原本的終端機視窗還開著上一個專案的對話，**不要用那一個**，開一個新的終端機視窗）

**✅ 自我檢查點**：確認這個新對話是「乾淨的」——你可以先問它一句：「你知道我在做什麼專案嗎？」如果它答不出來（或用猜的），代表環境是乾淨的，可以開始測試。

---

## 2. 執行測試：只餵它交接文件（15 分鐘）

把你 Session B4 寫好的交接文件**完整複製**，貼給這個全新對話，加上這句指令：

```
以下是一份專案的交接文件，我是剛接手這個專案的人，完全沒有參與過之前的討論。
請你讀完這份文件後，幫我延伸做一個新功能：〔挑一個你的專案裡還沒做的小功能〕。
如果文件裡有任何你看不懂、需要我補充的地方，請直接告訴我。

【交接文件內容】
（貼上你完整的交接文件）
```

**觀察重點**：
- 它有沒有回你「哪裡看不懂、需要你補充」？如果有，那就是文件真正的破口
- 它做出來的東西，跟你原本想的一不一樣？

---

## 3. 判讀結果：對照回文件的哪一節（15 分鐘）

把測試中出現的每一個疑問或落差，對照回交接文件的 10 節，找出問題出在哪：

| 測試中出現的狀況 | 通常對應到文件的哪一節沒寫清楚 |
|---|---|
| 不確定這個功能是「為誰做的」 | 第 2 節 Customer Needs |
| 不確定「做到什麼程度算做完」 | 第 4 節 Open Spec 的完成的定義 |
| 不確定資料該怎麼存、欄位是什麼 | 第 6 節 Schema |
| 做出來的東西跟你之前用 Claude Code 做的原型對不起來 | 第 7 節 Vibe Coding Log 沒寫清楚之前已經做到哪裡 |
| 不知道現在到底做完了沒、還缺什麼 | 第 8 節 目前完成度 |

**回頭修正**：針對測試中暴露出來的每一個破口，回去交接文件裡把對應章節補得更具體，然後**可以重複這個測試**，直到全新對話不太需要再追問為止。

---

## 4. 找碴練習：這是真的破口，還是合理的追問？

不是所有追問都代表文件有問題——要分辨：

| 狀況 | 判斷 |
|---|---|
| 追問「這個功能要不要支援中英文？」，但文件裡的 Constraint 明明已經寫了「中英文皆可提問」 | ⚠️ 它沒有讀仔細，重新明確引導它讀哪一節 |
| 追問「這個功能要不要支援中英文？」，文件裡完全沒提到語言需求 | ❌ 真的是文件漏寫了，回頭補 |
| 追問「你想要這個報表用長條圖還是折線圖？」，這是文件原本就沒打算規定的細節 | ✅ 合理的實作細節詢問，不算文件破口，正常回答它即可 |

- [ ] 我至少修正過一輪文件內容，並且理解修正的理由
