# Session B3 示範腳本：多重代理人（Multi-Agent）協作做法

> 對應：[工作坊設計文件](工作坊設計_AI-native產品開發工作坊.md) Session B3（60 分鐘）
> 承接：[SessionB2示範腳本_EMI案例.md](SessionB2示範腳本_EMI案例.md) 的駱世民教授（Prof. Shihmin Lo）案例——本節示範如何把同一個專案，拆給多個子代理人（Subagent）同時處理，並示範「哪些工作可以平行、哪些要排隊」的判斷方式。

---

## 0. 使用說明（給講師）

- 本腳本承接 Session B2 產出的第一版原型（課程頁面 + 雙語 AI 助教問答，已對照過 Definition of Done）。
- 「講師說」是建議台詞；「Claude Code 可能的回應」為模擬示意內容，**請事先實際操作過一次再上場**。
- 本節的教學重點不是「代理人怎麼運作」的技術細節，而是**老師（PjM）該怎麼判斷哪些工作可以平行、哪些要排隊**——這個判斷力比操作方法本身更重要。

---

## 1. 案例現況與今天的任務（回顧）

Session B2 已經做出第一版原型，涵蓋 S1（課程頁面）與部分 S2、S5（AI 助教問答）。今天要把剩下的工作做完整，並練習用多重代理人加快速度：

| 任務代號 | 對應使用者故事 | 內容 |
|---|---|---|
| 任務 A | S1 | 課程 16 週大綱與教材列表頁面 |
| 任務 B | S2、S5 | 雙語 AI 助教問答邏輯（含「信心不足要明確告知」） |
| 任務 C | S3、S4 | 對話紀錄儲存 + 每週卡關報表 |

---

## 2. 逐分鐘示範流程（60 分鐘）

### 段落 1（0–8 分鐘）承接與今天目標
**講師說**：
> 「Session B2 我們已經做出第一版原型，但只顧到課程頁面跟基本問答。今天要把剩下的功能做完整，而且要試試看：與其一件一件跟 Claude Code 講，能不能請它同時派人處理好幾件事，加快速度。」

---

### 段落 2（8–20 分鐘）判斷任務：哪些可以平行、哪些要排隊

**講師提問全班**：
> 「任務 A（課程頁面）、任務 B（AI 助教問答）、任務 C（對話紀錄 + 報表），這三個哪些可以同時做，哪些一定要等別的先做完？」

**帶回 Session A4 畫好的 Graphiti 關聯圖，一起看**：
```
Course
 ├── Module → Material
 ├── Student → Conversation
 ├── AI Coach → Conversation
 └── WeeklyReport（依 Conversation 統計而成）
```

👉 **核心教學重點**：WeeklyReport 這一條線，畫的時候就已經標明它是「依 Conversation 統計而成」——這代表任務 C（報表）**必須等任務 B（AI 助教問答，會產生 Conversation 資料）的資料格式確定之後才能開始**，不能跟 A、B 同時平行做。Graphiti 關聯圖不只是用來設計資料的工具，這時候直接變成判斷「哪些工作可以平行、哪些要排隊」的依據。

**結論（板書或投影）**：
- 任務 A 與任務 B：彼此獨立，可以平行
- 任務 C：依賴任務 B 的產出，要排在 B 之後

---

### 段落 3（20–35 分鐘）示範發派指令：平行處理 A 與 B

講師現場對 Claude Code 下達（可直接複製使用）：

```
接下來我要同時處理兩件事，這兩件事彼此不影響：

(A) 把課程 16 週大綱與教材列表做成一個獨立的頁面區塊。

(B) 把雙語（中/英）AI 助教問答邏輯做出來，重點是：
    - 學生可以用中文或英文提問，AI 用同樣的語言回覆
    - 當問題超出教材範圍、AI 沒有把握時，必須明確回覆
      「這個問題教材沒有涵蓋，建議詢問老師」，不能自己編答案

請你分別派人同時處理這兩件事，兩邊都做完後跟我回報結果，
並讓我知道 (B) 產出的問答紀錄，資料格式大概長什麼樣子。
```

**講師講解 Claude Code 通常會怎麼做**：
1. 把工作拆成兩條線，分別交給負責 A、負責 B 的處理程序
2. 兩邊背景執行，過程中老師可以繼續做別的事（例如檢查 Session B2 的 Vibe Coding Log），不需要一直盯著看
3. 兩邊都完成後，彙整回報：A 做了什麼、B 做了什麼，並說明 B 產出的對話資料格式（例如：問題、回覆、語言、是否為「超出範圍」的紀錄）

**提醒學員注意**：講師特意在指令最後加了一句「讓我知道 (B) 產出的資料格式」——這是為了替段落 5 的任務 C 做鋪陳，不是隨口問的。

---

### 段落 4（35–45 分鐘）觀察產出 + 派一個審查型代理人交叉檢查

A、B 完成後，講師下達審查指令：

```
現在請你再派一個人，專門檢查剛才 A 跟 B 做出來的東西，
有沒有符合以下的完成的定義：

S1：首頁列出 16 週課程大綱，每週有對應教材下載連結
S2：聊天視窗可輸入問題，AI 回覆需引用該週教材內容，中英文皆可提問並以提問語言回覆
S5：問題超出教材範圍或信心不足時，AI 必須明確回覆「教材沒有涵蓋，建議詢問老師」

如果有沒做到的地方，請列出來，不要直接動手改，先讓我看清單。
```

**教學重點**：
- 讓「做的人」跟「檢查的人」分開，跟工程團隊「寫程式的人不review自己的程式」是同一個道理
- 先列清單再決定要不要改，讓老師（PjM）保有最後決定權，而不是任由 AI 自己全部改掉

**對照檢查表**（投影，帶全班一起看審查結果）：

| 編號 | 審查型代理人回報 | 老師決定 |
|---|---|---|
| S1 |  | 通過 / 請 A 修正 |
| S2 |  | 通過 / 請 B 修正 |
| S5 |  | 通過 / 請 B 修正 |

---

### 段落 5（45–53 分鐘）示範相依任務：報表（任務 C）要等 B 確定後才開始

**講師說**：
> 「現在 B 的問答資料格式已經確定了，我們才可以開始做任務 C——因為報表是統計這些問答資料做出來的，順序不能顛倒。」

下達指令：

```
現在請根據剛才 (B) 產出的問答資料格式，開始做任務 C：
把每一筆問答紀錄依主題分類、統計次數，做出一個「本週最常被問三個主題」的報表頁面，
給老師端查看。
```

**帶討論**：
> 「如果我們一開始不管相依關係，把 A、B、C 三個同時發派下去，可能會發生什麼事？」

引導答案：C 可能會先假設一種資料格式去做統計，等 B 真正做完後兩邊格式對不起來，還要重工——**呼應「不要為拆而拆」，多重代理人不是拆得越多越快，要先判斷相依關係**。

---

### 段落 6（53–60 分鐘）各組操作 + 收尾

**活動**：各組回到自己的專案，用自己畫的 Graphiti 關聯圖，判斷手上的工作項目哪些可以平行、哪些要排隊，並用今天示範的句型請 Claude Code 分工處理至少一組平行任務。

**收尾提醒**：
- 在 Vibe Coding Log 的「分工代理人」欄位，把今天 A / B / C 分別記錄下來
- 講師預告 Session B4：接下來要把今天的成果整理進交接文件，讓 TA 或協作教師就算沒參與今天的過程，也能看懂「哪些任務誰做的、目前做到哪裡」

## 3. 常見狀況與講師應對（FAQ）

| 狀況 | 建議應對 |
|---|---|
| 兩個代理人同時修改同一批檔案，會不會互相覆蓋？ | 讓 Claude Code 自己處理工作區隔離即可，講師不需要向學員解釋底層機制，只需要知道有這個機制存在 |
| 學員想把所有工作都拆給不同代理人，即使是很小的項目 | 提醒：過度拆分會增加协调成本，應該先判斷「真的彼此獨立」且「量體夠大」才值得拆，小專案一個對話就處理得完 |
| 審查型代理人檢查出問題，要直接讓它改，還是先看清單？ | 建議先看清單，由老師（PjM）決定要不要改、怎麼改，保留最後判斷權，而不是全部交給 AI 自行決定 |
| 學員忘記考慮相依關係，把 C 跟 A、B 一起發派 | 帶回 Graphiti 關聯圖，指出哪一條線代表「誰依賴誰」，這通常就是判斷順序的依據 |
